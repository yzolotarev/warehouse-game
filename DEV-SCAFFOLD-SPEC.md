# Dev-Scaffold: агентский таск-менеджмент проекта через хуки

> Итог мета-сессии 14.07.2026 (Ярослав + Claude). Durable-спека для реализации.
> Принцип: детерминированный scaffolding, механика вместо памяти модели, работает
> при минимуме токенов (аналог «работает в худшем состоянии человека»).

## Главный инсайт: это ОДНА система с 6 сторон

Все нитки разговора схлопываются в один per-project движок **`dev-warehouse`** =
склад задач агента при реализации проекта. Это движок warehouse-game **минус
игровой слой** (агенту не нужны очки/анимации/один-объект-на-экран - у него нет
усталости и экрана), в обёртке headless-CLI.

```
              per-project движок "dev-warehouse"
   ┌──────────────────────────────────────────────────┐
   │ inbox ──► triage(разбор) ──► radar/focus          │
   │   ▲            │                                   │
   │   │            └─► опц. ds-audit "make it clean"   │
   │ капчур мыслей         (внешнее мнение нейронки)    │
   └──────────────────────────────────────────────────┘
        △ дёргается стопором = Stop-хук
```

## Компоненты и решения

### 1. Радар проекта (source of truth)
- Один файл на проект (правило «один файл»). НЕ новый склад: переиспользовать слой
  `.dev` / `current_focus.json` из скилла `init`. Радар = честная витрина: делаю /
  готово / дальше / затык.

### 2. Впрыск (чтение) = SessionStart-хук
- Клон `graphify-notify.sh`: смотрит cwd → узнаёт проект → впрыскивает радар.
  Проверенный паттерн, ~час работы. Обязательное чтение, как graphify-out.

### 3. Запись назад = Stop-хук (стопор). Несущая стена, НЕ украшение.
- Пассивный хук писать не умеет - запись требует, чтобы агент оценил прогресс.
  Поэтому после хода Stop-хук МЕХАНИЧЕСКИ впрыскивает: «сделал шаг? обнови радар».
- Насаживание, как auto-qualify («модель не забывает») и crash-logger.
- Из DESIGN-LOG склада: убери обязательный разбор → развал за 2 недели. Значит
  ревью обязано дёргаться хуком, а не волей. Иначе радар протухнет.
- Развилка (решить при стройке): триггер тяжёлого макро-ревью = счётчик ходов
  (рекоменд.) + ручная команда `/checkpoint`.

### 4. Триаж-дерево агентского инбокса = КОПИЯ канонического
- Не изобретать: взять GTD-веер warehouse (что это → делать → мне → сейчас →
  мутная=проект → ≤2мин=сразу). Правило склада «дерево разбора не трогать» → у
  Ярослава одна модель в голове, не две.
- Старение (пыль 3дня / заморозка 30дн) и стопор переносятся из движка.

### 5. Внешнее ревью = ds-audit (ПОЧИНИТЬ)
- `ds-audit` сломан: скилл жёстко зашит на legacy `deepseek-cli/server_playwright.py`
  (не запущен, throwaway-браузер, риск бана). Заменён v2 `deepseek-cli-anything`:
  команда `ds` (CDP к реальному Chrome :9222, cli-anything-методология).
- ФИКС: перенаправить скилл `~/.claude/skills/ds-audit/SKILL.md` с `server_playwright.py`
  на `ds` v2. Скорее всего это весь баг. Chrome :9222 уже открыт.

### 6. Слой мыслей = НЕ авто-граф (ловушка, отклонено)
- Авто «граф мыслей» = дешёвые узлы И дешёвые стрелки = галлюцинация головы юзера.
  Противоречит аксиоме: узлы дёшевы (ИИ), стрелки священны (человек).
- graphify уже ест `.md` (даёт rationale-рёбра), wiki-build = слой решений. Третий
  авто-граф = оверхед + дубль. НЕ строить.
- Настоящий слой мыслей = стрелки, которые Ярослав рисует на стопор-ревью (mindmap /
  когнитивный обмен). ИИ готовит холст узлов из graphify/wiki, человек тянет 1-2
  стрелки. Одна честная стрелка > 100 авто-сгенерированных.

### 7. Роутинг мыслей (новая идея 14.07) - фильтр на входе НЕТ
- Проблема: мысль приходит днём, тратить 30с на поиск нужной папки проекта нельзя.
- Решение в духе склада («фильтра на входе нет, чистка на триаже»): мысль падает в
  ОДИН инбокс без выбора проекта. Роутинг - на стопор-ревью: агент проекта X сам
  «забирает» из общего инбокса то, что относится к X (авто-матч по проекту, юзер
  подтверждает одной клавишей). Один рот, N project-claims на ревью.
- Капчур уже есть: `warehouse/capture/capture.sh` → POST `:8091/inbox` (поле `source`).
  Расширение: опц. поле `project`; если пусто - матчится на триаже.
